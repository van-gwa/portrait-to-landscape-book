# Lint 工作流：Wiki 健康检查

## 📖 目的
定期清理和验证 Wiki，确保所有内容质量高、一致性强、没有孤立或破损内容。

---

## 🎯 流程概览

```
启动定期检查（周/月）
   ↓
[第 1 步] 扫描断裂链接 ← 找出指向不存在的页面的链接
   ↓
[第 2 步] 发现孤立页面 ← 找无任何传入或传出链接的页面
   ↓
[第 3 步] 检查矛盾 ← 发现新旧信息的冲突
   ↓
[第 4 步] 验证标签 ← 确保分类规范
   ↓
[第 5 步] 更新过期内容 ← 标记或更新超过 3 个月未改的页面
   ↓
[第 6 步] 生成报告 ← 输出改进清单
   ↓
✅ 完成：Wiki 已清理，质量得到保证
```

---

## 📋 详细步骤

### 第 1 步：扫描断裂链接

**任务**：找出所有指向不存在页面的链接

**操作**：
```
在 Obsidian 中：
1. 打开"未链接的文件"（Outline 面板）
2. 检查是否有指向不存在的页面的链接
3. 使用正则表达搜索：\[\[.*?\]\] 来找所有 Wiki 链接

或者使用 Dataview：
查询所有包含 Wiki 链接的页面，逐个验证
```

**修复策略**：

| 情况 | 处理 |
|------|------|
| 链接指向应该存在的页面，但页面缺失 | 创建该页面（优先级：高） |
| 链接指向已删除的页面 | 删除链接或替换为新链接 |
| 链接拼写错误 | 纠正拼写 |
| 链接指向过期的概念 | 更新为指向新概念，添加"已更新"说明 |

**修复记录**：
```markdown
## Lint 修复记录 - 2026-05-XX

### 断裂链接修复（共 X 个）
- [[old-broken-link]] → [[new-correct-link]]
- [[missing-page]] → 已创建页面
- [[typo-link]] → [[correct-link]]

### 创建的新页面（共 X 个）
- [[new-page-1]]
- [[new-page-2]]
```

---

### 第 2 步：发现孤立页面

**任务**：找出没有任何传入或传出链接的页面（孤立岛）

**识别标准**：
```
孤立页面 = 
  (传入链接数 = 0) AND (传出链接数 = 0)
```

**操作**：
```
使用 Dataview 查询：
```dataview
TABLE file.inlinks.length as "传入链接", file.outlinks.length as "传出链接"
WHERE file.inlinks.length = 0 AND file.outlinks.length = 0
```

或手动检查 Graph View：
1. 打开 Graph View
2. 查找独立的节点（没有连线）
```

**处理策略**：

| 情况 | 处理 |
|------|------|
| 页面重要但孤立 | 添加 3-5 个相关链接 |
| 页面过时无用 | 标记 `#deprecated`，考虑删除 |
| 页面是独立指南 | 将其添加到导航索引 |
| 页面尚未完成 | 标记 `#draft`，添加到待办清单 |

**优先级判断**：
```
高优先级：
- 核心概念页面（应该有大量链接）
- 实体页面（应该被其他页面引用）

中优先级：
- 指南页面（可以有较少链接）
- 旧时期的实验页面

低优先级：
- 临时笔记
- 已标记过期的页面
```

---

### 第 3 步：检查矛盾

**任务**：发现同一概念在不同页面中的定义或描述矛盾

**操作**：
```
对于关键概念（如 UI 布局类型）：
1. 搜索该概念在 Wiki 中出现的所有位置
2. 比较定义和描述
3. 检查引用的数据是否一致
4. 寻找前后不一致的地方
```

**检查清单**：
- [ ] UI 布局类型的定义是否一致？
- [ ] 组件属性的描述是否一致？
- [ ] 统计数字是否匹配（如 UI 数量）？
- [ ] 相关关系的描述是否对称？

**常见矛盾类型**：

```
类型 1：定义不一致
  页面 A：[定义 1]
  页面 B：[定义 2]
  → 哪个正确？需要澄清

类型 2：数据不匹配
  页面 A：四边框架布局有 47 个 UI
  页面 B：四边框架布局有 45 个 UI
  → 哪个是最新的？需要验证

类型 3：逻辑冲突
  页面 A：组件 X 使用于 UI 类型 Y
  页面 B：UI 类型 Y 不包含组件 X
  → 需要调和
```

**修复步骤**：
```
1. 标记矛盾页面为 #needs-verification
2. 在页面中添加"矛盾说明"部分
3. 链接到相关页面并说明矛盾内容
4. 后续（可能需要手动验证）：
   - 检查源数据
   - 更新错误的信息
   - 移除 #needs-verification 标签
```

---

### 第 4 步：验证标签规范

**任务**：确保所有页面的标签遵循规范

**检查标准**：
```
✅ 正确做法：
- 每个页面有 2-5 个标签
- 标签遵循规范（#concept, #layout-type-*, #needs-update 等）
- 标签拼写正确

❌ 常见错误：
- 标签为空（#empty）
- 标签过多（>5 个）
- 标签命名不规范（#myTag, #random_tag）
- 标签拼写错误（#need-update vs #needs-update）
```

**操作**：
```
按类型检查：

1. 概念页面应该有：
   #concept, #[topic-name], [可选：优先级]

2. 实体页面应该有：
   #entity, #[category], #[type-specific], [可选：状态]

3. 指南页面应该有：
   #guide, #[topic-name], #[status], [可选：优先级]

4. 所有页面可能有：
   #high-priority, #needs-update, #needs-verification, #incomplete
```

**修复**：
```
- [ ] 添加缺失的标签
- [ ] 纠正拼写错误的标签
- [ ] 移除重复标签
- [ ] 重新组织标签（不超过 5 个）
```

---

### 第 5 步：更新过期内容

**任务**：标记或更新超过 3 个月未修改的页面

**识别过期页面**：
```
过期 = (当前日期 - updated 日期) > 90 天
```

**操作**：
```
1. 列出所有超过 90 天未更新的页面
2. 按优先级排序（核心概念 > 常用实体 > 罕见内容）
3. 对每个页面决策：
```

**修复决策**：

| 页面优先级 | 信息可能过时？ | 操作 |
|-----------|-----------|------|
| 高 | 是 | 立即审查并更新 |
| 高 | 否 | 更新"updated"日期 |
| 中 | 是 | 标记 #needs-update，后续处理 |
| 中 | 否 | 标记为已检查 |
| 低 | 任何 | 考虑标记 #archived |

**标记过期内容**：
```markdown
## ⚠️ 更新说明
最后验证：2026-05-04
下次应审查：2026-08-04
已检查内容准确性：是/否
```

---

### 第 6 步：生成 Lint 报告

**任务**：输出总结报告和改进清单

**报告模板**：

```markdown
# Lint 报告 - 2026-05-XX

## 📊 整体统计
- 总页面数：[N]
- 断裂链接：[N] 个（已修复 X 个）
- 孤立页面：[N] 个
- 发现矛盾：[N] 处
- 标签不规范：[N] 页
- 超期未更新：[N] 页

## 🔧 已完成修复
- [修复 1]
- [修复 2]
- ...

## 🚨 需要手动处理
### 断裂链接（需要创建页面）
- [ ] [[missing-page-1]]
- [ ] [[missing-page-2]]

### 孤立页面（需要链接或删除）
- [ ] [[orphan-page-1]] - 建议操作：[action]
- [ ] [[orphan-page-2]] - 建议操作：[action]

### 矛盾内容（需要澄清）
- [ ] [[page-1]] vs [[page-2]] - 矛盾：[description]

### 标签问题
- [ ] [[page-1]] - 缺少标签
- [ ] [[page-2]] - 标签拼写：[old] → [new]

### 过期内容
- [ ] [[page-1]] - 上次更新：2026-02-01（需要审查）

## 📈 Wiki 健康度评分
- 完整性：[N]%（有多少页面完整？）
- 一致性：[N]%（有多少概念定义一致？）
- 可维护性：[N]%（有多少页面有明确的关键字？）
- **整体评分：[N]/100**

## 📅 后续行动计划
- [ ] 修复前 3 个断裂链接（预计 1 小时）
- [ ] 处理孤立页面（预计 2 小时）
- [ ] 审查并更新 5 个高优先级过期页面（预计 1 小时）
- [ ] 协调矛盾内容（预计 1 小时）

**预计总工作量**：[N] 小时
**下次 Lint 计划**：2026-06-04 或当有重大更新后
```

## 🔄 Lint 周期计划

### 推荐频率
```
轻量 Lint（15-30 分钟）：每周一次
  - 扫描断裂链接
  - 快速标签检查
  - 标记新增的孤立页面

完整 Lint（2-4 小时）：每月一次
  - 所有 6 个步骤完整执行
  - 生成完整报告
  - 计划后续改进

深度 Lint（半天）：每季度一次
  - 不仅检查形式，也审查内容准确性
  - 检查概念的演进是否反映在文档中
  - 重新评估分类和组织结构
```

---

## 📌 Lint 最佳实践

### 何时应该运行 Lint
✅ 大量新内容添加后（>20 页）
✅ 发现多个用户报告的错误后
✅ 定期计划的 Lint 周期到期
✅ 在重大重构前（用 Lint 报告作为基线）

### 何时不用运行完整 Lint
❌ 只做一个小修改（可以跳过）
❌ 紧急时期（可以延后）
✅ **但建议至少做轻量扫描（5 分钟）**

### Lint 的自动化
```
可以写脚本自动化的部分：
- 扫描断裂链接（脚本检测 [[...]] 的有效性）
- 计算孤立页面（脚本统计链接数）
- 标签重复检查（脚本发现重复标签）
- 生成基础报告（脚本统计数据）

需要人工的部分：
- 判断矛盾的真实含义
- 决定孤立页面是否应该保留
- 评估过期内容是否还有价值
- 更新内容保证准确性
```

```
