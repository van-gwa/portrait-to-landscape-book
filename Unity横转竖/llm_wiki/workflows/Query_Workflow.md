# Query 工作流：查询和合成答案

## 📖 目的
当用户提出问题时，通过 **Query** 流程在 Wiki 中搜索相关信息，综合答案，并将有价值的发现转化为新的 Wiki 页面。

---

## 🎯 流程概览

```
用户问题
   ↓
[第 1 步] 理解问题 ← 解析关键词和概念
   ↓
[第 2 步] 搜索 Wiki ← 找到相关页面
   ↓
[第 3 步] 综合信息 ← 整合多个页面的内容
   ↓
[第 4 步] 生成答案 ← 清晰的回复
   ↓
[第 5 步] 更新 Wiki ← 将新发现转化为页面
   ↓
✅ 完成：问题已解答，Wiki 已充实
```

---

## 📋 详细步骤

### 第 1 步：理解问题

**任务**：解析用户问题，提取关键词和隐含需求

**操作**：
```
- [ ] 识别问题的核心主题（UI、布局、组件、优化等）
- [ ] 列出关键词和概念（需要搜索的 Wiki 页面）
- [ ] 确定问题的层级（概念性、实践性、对比性）
- [ ] 明确期望的答案形式（列表、分析、决策建议等）
```

**检查清单**：
- 问题的主要实体是什么？（UI、布局、组件）
- 问题的信息需求是什么？（定义、统计、对比）
- 是否包含隐含的后续问题？

**问题分类**：
```
类型 1：定义性问题
  问：什么是 [概念]？
  操作：搜索 [[concept-page]]
  
类型 2：实体查询
  问：[实体] 有什么属性？
  操作：搜索 [[entity-page]]
  
类型 3：关系问题
  问：[概念A] 和 [概念B] 有什么关系？
  操作：搜索两个概念页面，分析关联
  
类型 4：统计问题
  问：有多少个 [实体]？
  操作：搜索索引页面，聚合数据
  
类型 5：决策问题
  问：我应该如何处理 [情景]？
  操作：搜索指南页面，综合建议
```

---

### 第 2 步：搜索 Wiki

**任务**：在 Wiki 中定位相关页面

**操作**：
```
使用 Obsidian 搜索功能（Ctrl+Shift+F）：

搜索策略：
1. 直接搜索关键词
2. 浏览相关索引页面（Index_*）
3. 沿着反向链接探索关联内容
4. 检查标签集合（#layout-type-*, #component-*）
```

**确定优先级**：
```
第一优先：直接匹配的概念或实体页面
第二优先：相关的指南页面
第三优先：索引页面（用于导航）
第四优先：具有反向链接的页面
```

**搜索结果评估**：
- [ ] 找到了核心概念页面？
- [ ] 找到了相关实体页面？
- [ ] 是否有对比页面？
- [ ] 索引页面是否完整？

**缺失页面处理**：
如果关键页面缺失：
```
标记为需要创建的新页面：
- 页面名称：[name]
- 应属类型：concept/entity/guide
- 优先级：high/medium/low
- 原因：[为什么这个页面重要]
```

---

### 第 3 步：综合信息

**任务**：整合多个 Wiki 页面的信息，生成连贯的答案

**操作**：
```
1. 列出所有相关页面的关键信息点
2. 识别信息之间的关系和层级
3. 查找矛盾或需要澄清的地方
4. 组织信息结构
```

**信息组织模板**：
```
核心概念：[概念定义，来自 concept 页面]
  ├─ 相关属性：[从 entity 页面]
  ├─ 应用场景：[从 guide 页面]
  ├─ 相关概念：[交叉引用]
  └─ 数据统计：[从 index 页面]

对比分析：[如适用]
  ├─ 选项 A：[来自页面 A]
  └─ 选项 B：[来自页面 B]
```

**质量检查**：
- [ ] 信息是否完整？
- [ ] 逻辑是否清晰？
- [ ] 是否有矛盾之处需要调和？
- [ ] 是否有过时信息需要标记？

---

### 第 4 步：生成答案

**任务**：用清晰的语言向用户解答问题

**答案结构**：
```markdown
## 简单回答
[一句话总结]

## 详细解释
[深入说明]

## 相关资源
- [[wiki-page-1]]
- [[wiki-page-2]]

## 进一步探索
[相关问题或延伸阅读]
```

**答案质量检查**：
- [ ] 是否直接回答了问题？
- [ ] 是否逻辑清晰、结构合理？
- [ ] 是否引用了关键 Wiki 页面？
- [ ] 是否避免了不必要的技术术语？
- [ ] 是否包含了相关资源的链接？

---

### 第 5 步：更新 Wiki

**任务**：将查询过程中的发现和学习转化为 Wiki 改进

**更新内容**：

#### 5.1 发现新的连接点
```
如果发现两个页面间有新的关系：
→ 在双方页面中添加反向链接
→ 更新 "related" 字段
```

#### 5.2 完善现有页面
```
如果现有页面需要补充信息：
→ 添加新的示例、应用场景、数据
→ 更新 frontmatter 的 "updated" 日期
→ 在"最近发现"部分记录补充
```

#### 5.3 创建新概念页面
```
如果发现需要创建新概念页面：
→ 立即创建页面（使用 Ingest 工作流）
→ 或标记为 #needs-wiki-entry（后续批量处理）
```

#### 5.4 标记需要验证的矛盾
```
如果发现信息矛盾：
→ 在相关页面标记 #needs-verification
→ 在页面中记录矛盾的具体内容
→ 加入"待验证问题"清单
```

#### 5.5 更新索引页面
```
如果查询涉及多个同类实体：
→ 检查索引页面是否完整
→ 添加缺失的链接
→ 更新统计数字
```

## 💡 实战示例

### 场景：用户问"哪些 UI 使用了四边框架布局"

**Query 过程**：

```
[第 1 步] 理解问题
✓ 关键词：UI 列表、布局类型 "四边框架"
✓ 问题类型：统计 + 实体列表查询
✓ 期望答案：UI 名称列表 + 数量

[第 2 步] 搜索 Wiki
✓ 搜索：#layout-type-four-border
✓ 发现：Index_Layout_Types 页面
✓ 发现：对应的布局概念页面

[第 3 步] 综合信息
✓ 从索引收集所有 UI 链接
✓ 验证每个 UI 的布局标签
✓ 统计总数：47 个 UI

[第 4 步] 生成答案
"四边框架布局（Four Border Frame）用于 47 个 UI...
详见 [[Index_Layout_Types]] 或 [[concept/four-border-frame]]
"

[第 5 步] 更新 Wiki
✓ 检查索引页面数据是否最新
✓ 在布局概念页面添加"查询统计"部分
✓ 标记任何缺失的 UI 链接
✓ 更新统计时间戳
```

## 📊 查询统计和模式识别

### 记录查询信息
每次成功的查询都应该记录：
```yaml
查询日期: 2026-05-04
问题类型: [统计/定义/对比/决策]
相关页面: [[page1]], [[page2]]
发现缺陷: [如有]
新页面创建: [如有]
```

### 优化索引
定期分析：
- 最常被查询的概念？→ 优先充实这些页面
- 最常产生的后续问题？→ 创建指南页面
- 最常需要补充的信息？→ 改进现有页面

## ⏱️ 预计时间

| 步骤 | 时间 |
|------|------|
| 第 1 步：理解 | 2-5 分钟 |
| 第 2 步：搜索 | 3-10 分钟 |
| 第 3 步：综合 | 5-15 分钟 |
| 第 4 步：生成答案 | 5-15 分钟 |
| 第 5 步：更新 Wiki | 10-30 分钟 |
| **总计** | **25-75 分钟** |

## 📌 最佳实践

### 何时深入挖掘
```
✅ 问题涉及多个相关概念
✅ 答案可能对其他用户有价值
✅ 发现了新的 Wiki 缺陷
```

### 何时快速回答
```
✅ 问题可直接从一个页面回答
✅ 没有发现新的信息缺口
✅ 用户时间紧急
```

### 常见陷阱
❌ **不要**：仅回答问题而忘记更新 Wiki
✅ **应该**：总是标记至少一个改进点

❌ **不要**：在 Wiki 中记录过于详细的查询日志
✅ **应该**：记录关键发现和改进
