# UI 分类检查执行日志系统 — 完整指南

## 📋 概述

**执行日志系统**是自动化 UI 分类检查系统的详细记录层，用于记录每次检查的具体数据。

```
一句话总结：每次检查都会自动生成一条详细的执行记录。
```

## 🔄 数据流

```
【每次检查】

1. Claude 执行检查
   ├─ 选择随机 UI 图片
   ├─ 分析图片判断类型
   ├─ 查询已有分类
   └─ 对比结果
   
2. 获取当前时间和日期
   ├─ 当前时间：HH:mm:ss
   └─ 当前日期：YYYY-MM-DD

3. 打开当天的执行日志
   ├─ 路径：execution_logs/2026-05-06.md
   └─ 如不存在则自动创建

4. 在执行记录表末尾添加新行
   ├─ 时间、UI 名字、初判类型、实际类型
   ├─ 结果（✅/⚠️/❌）
   └─ 备注

5. 更新每日统计
   ├─ 总检查数 +1
   ├─ 通过数 +1（如果通过）
   ├─ 问题数 +1（如果有问题）
   └─ 更新统计表

6. 如有问题，同时添加到问题记录表
   ├─ 记录时间、UI、类型对比
   └─ 记录问题类型

7. 更新"最后更新"时间戳

【结果】
✅ 日志记录完成
```

## 🔧 技术细节

### 自动创建机制

```
系统会在首次检查时执行：

1. 获取当前日期（如 2026-05-06）
2. 检查文件是否存在：
   execution_logs/2026-05-06.md
3. 如不存在：
   ├─ 读取模板：execution_logs/_TEMPLATE.md
   ├─ 替换日期和时间占位符
   └─ 创建新文件
4. 如已存在：
   └─ 直接打开，追加新数据
```

### 数据追加规则

```
每次检查完成后：

1. 打开当天的日志文件
2. 找到"检查执行记录"表格
3. 在表格末尾（在最后一行下方）添加新行
4. 新行数据格式：
   | HH:mm:ss | UI_NAME | TYPE_A | TYPE_B | ✅/⚠️/❌ | 备注 |
5. 更新"每日统计"表格中的数字
6. 保存文件
```

### 时间戳格式

```
执行时间：HH:mm:ss（24小时制）
  示例：08:35:42、14:23:15、23:59:59

创建/更新日期：YYYY-MM-DD
  示例：2026-05-06

完整时间戳：YYYY-MM-DD HH:mm:ss
  示例：2026-05-06 08:35:42
```

## 📊 性能优化

### 文件分割策略

```
按天分割：
- 优点：每个文件大小小（15-25 KB）
- 优点：易于查询和管理
- 优点：避免单个文件过大
- 缺点：文件数多（365 个/年）

理由：
  Obsidian 能轻松处理数百个 .md 文件
  按天分割更符合日志使用习惯
```

### 数据保留策略

```
当前：保留所有历史数据
  - 1 周数据：~140 KB
  - 1 个月数据：~600 KB
  - 1 年数据：~7.2 MB

可选：按月归档
  - 每月末压缩当月日志
  - 压缩率：约 50%
  - 节省空间但影响查询速度
```

## 🎯 常见问题

**Q: 为什么要每天一个文件？**  
A: 方便查询和管理。每个文件只有一天的数据，大小合理，易于定位信息。

**Q: 日志文件会不会太多？**  
A: 不会。365 个文件总大小仅 ~7 MB，Obsidian 能轻松处理。

**Q: 如何快速找到某个 UI 的检查记录？**  
A: 使用 Obsidian 的全局搜索（Ctrl+Shift+F）搜索 UI 名称。

**Q: 统计数字是自动更新吗？**  
A: 是的。系统自动计算和更新，无需手动修改。

**Q: 旧日志会保留吗？**  
A: 是的。所有历史日志都会保留作为参考和分析。

**Q: 可以导出数据吗？**  
A: 可以。日志是纯 Markdown 文本，可轻松导出为 CSV、Excel 等格式。
