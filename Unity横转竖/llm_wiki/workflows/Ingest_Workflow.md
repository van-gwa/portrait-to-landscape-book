# Ingest 工作流：处理新源文档

## 📖 目的
当获得新的源文档或数据时，LLM 通过 **Ingest** 流程将其分解、总结，并集成到 Wiki 知识库中。

---

## 🎯 流程概览

```
新源文档
   ↓
[第 1 步] 阅读并理解 ← 提取关键概念和实体
   ↓
[第 2 步] 映射概念 ← 识别现有 Wiki 页面的连接点
   ↓
[第 3 步] 创建/更新 ← 新建页面或修改现有页面
   ↓
[第 4 步] 交叉引用 ← 建立页面间的链接
   ↓
[第 5 步] 验证 ← 检查一致性和完整性
   ↓
✅ 完成：新内容已整合到 Wiki
```

---

## 📋 详细步骤

### 第 1 步：阅读与分析源文档

**任务**：理解新文档包含的信息

**操作**：
```
- [ ] 完整阅读源文档
- [ ] 标记关键段落（概念、数据、规则）
- [ ] 列出 3-5 个主要要点
- [ ] 识别新信息 vs 已知信息
```

**检查清单**：
- 这个文档涵盖什么核心概念？
- 它改变或补充了哪些现有认知？
- 是否有矛盾或需要澄清的地方？

**输出**：
```
关键点总结：
1. [要点1]
2. [要点2]
3. [要点3]

新概念：[列表]
新实体：[列表]
需要验证的矛盾：[列表]
```

---

### 第 2 步：映射到现有 Wiki

**任务**：确定源文档与现有 Wiki 页面的关系

**操作**：
```
- [ ] 搜索现有相关页面（用搜索功能）
- [ ] 列出所有相关的 Wiki 页面
- [ ] 标记"需要新建"vs"需要更新"的页面
- [ ] 识别缺失的中间概念
```

**映射表模板**：
| 源文档内容 | 相关 Wiki 页面 | 操作 |
|-----------|-----------|------|
| 概念 A | [[existing-page]] | 更新 |
| 概念 B | 无 | 新建 |
| 实体 C | [[entity-page]] | 补充 |

**输出**：
- 需要新建的页面清单（5-15 个）
- 需要更新的页面清单
- 新概念依赖的旧概念

---

### 第 3 步：创建和更新页面

**任务**：生成或修改 Wiki 页面

**对于新页面**：
```markdown
根据 Schema 创建页面：
- 选择正确的类型（concept/entity/guide）
- 填充必需的 frontmatter
- 编写内容（参考 2.1-2.3 页面类型规范）
- 至少写 2-3 个段落
```

**对于更新现有页面**：
```markdown
- 在"最近发现"或"补充信息"部分添加新内容
- 更新"related"和"concepts"字段
- 更新 frontmatter 中的"updated"日期
- 不删除原有内容，只追加
```

**检查清单**：
- [ ] 页面有合法的文件名
- [ ] Frontmatter 字段完整
- [ ] 至少有 2 个出站链接
- [ ] 标签 2-5 个
- [ ] 代码块或示例（如需要）

---

### 第 4 步：建立交叉引用

**任务**：通过双向链接连接相关页面

**操作**：
```
对每个新建页面：
1. 在相关概念页面中添加反向链接
2. 在相关实体页面中添加"related"字段
3. 在索引页面中添加链接
4. 在指南页面中提及该页面（如适用）
```

**双向链接规则**：
- 如果 A → B，则 B 应该知道 A 的存在
- 使用 `related` 字段或正文中的 Wiki 链接
- 避免循环依赖（A→B→A）

**示例**：
```
页面 A（概念）：
- related: [[entity-uses-this-concept]]

页面 B（实体）：
- concepts: [[concept-page-a]]
```

---

### 第 5 步：验证和发布

**任务**：确保新内容符合质量标准

**验证清单**：
```
[ ] 没有断裂的 Wiki 链接（用搜索验证）
[ ] 所有引用的页面都存在
[ ] 标签遵循规范（#concept, #layout-type-*, 等）
[ ] 没有拼写错误或格式问题
[ ] 新概念与现有概念无矛盾（如有矛盾，记录在 #needs-verification）
[ ] 页面有明确的"最后更新"时间
```

**一致性检查**：
- 命名规范是否一致？
- 前后概念定义是否统一？
- 数据值是否匹配？

**发布前修复**：
如发现问题：
1. 标记为 `#needs-update` 或 `#needs-verification`
2. 立即修改错误信息
3. 在"最近发现"部分记录修正内容

## 💡 实战示例

### 场景：新增 UI 布局类型数据

**源文档**：新的 UI 布局分析 JSON 文件

**Ingest 过程**：

```
[第1步] 分析
✓ 发现 3 个新布局类型
✓ 发现 50 个新的 UI 与旧布局类型的映射
✓ 发现 2 个旧布局类型需要重新分类

[第2步] 映射
✓ 相关 Wiki 页面：
  - [[llm_wiki/wiki/concepts/UI_Layout]]
  - [[llm_wiki/wiki/indexes/Index_Layout_Types]]
  - 相关 UI 实体 (×50)

[第3步] 创建/更新
✓ 新建 3 个布局概念页面
✓ 更新 50 个 UI 实体页面
✓ 更新索引页面

[第4步] 交叉引用
✓ 在 Index 中添加新布局类型链接
✓ 在每个 UI 页面中标记其布局类型
✓ 在布局类型页面中列出相关 UI

[第5步] 验证
✓ 检查所有链接有效性
✓ 确保命名规范一致
✓ 验证数据准确性
✓ 标记需要进一步分析的项目
```

---

## ⏱️ 预计时间

| 步骤 | 时间 |
|------|------|
| 第 1 步：分析 | 5-15 分钟 |
| 第 2 步：映射 | 10-20 分钟 |
| 第 3 步：创建/更新 | 20-60 分钟（取决于页面数量） |
| 第 4 步：交叉引用 | 10-30 分钟 |
| 第 5 步：验证 | 10-15 分钟 |
| **总计** | **1-2 小时** （小规模内容）<br/>**半天** （大规模内容） |

## 📌 注意事项

### 常见陷阱
❌ **不要**：过度创建新页面（会导致碎片化）
✅ **应该**：考虑能否在现有页面中补充

❌ **不要**：忽视链接的创建
✅ **应该**：建立完整的双向链接

❌ **不要**：盲目相信新源文件正确性
✅ **应该**：与现有数据进行对照检查

### 何时创建新页面 vs 更新现有页面

**新建页面** 当：
- 概念完全新颖，无现有对应
- 实体之前完全未记录过
- 需要独立详细说明

**更新现有页面** 当：
- 信息补充或深化现有内容
- 数据修正或澄清
- 增加新应用场景或示例
