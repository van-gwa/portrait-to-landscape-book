## 📖 日志格式说明

### 执行日志的结构

每个日志文件（如 2026-05-06.md）包含：

1. **每日统计表**（自动更新）
   - 总检查数、通过数、问题数、通过率
   - 首次和最后的检查时间

2. **检查执行记录表**（逐行追加）
   - 执行时间、UI 名字、初判类型、实际类型、结果

3. **问题记录表**（只记录问题）
   - 问题时间、UI、判定错误、原因、建议

4. **每日汇总**
   - 检查开始和结束时间
   - 总耗时和平均耗时

### 更新机制

```
每次检查完成后：
1. 获取当天的日期（如 2026-05-06）
2. 打开对应的日志文件（execution_logs/2026-05-06.md）
3. 如果文件不存在，自动创建
4. 在检查执行记录表末尾添加新行
5. 更新每日统计数字
6. 如有问题，在问题记录表添加条目
```

## 💡 使用场景

### 场景 1：查看今天的检查情况

```
打开：[[execution_logs/2026-05-06]]

可以看到：
- 今天运行了多少次检查
- 有多少次通过，多少次出现问题
- 每次检查的具体 UI 和结果
```

### 场景 2：查看某个 UI 是否被检查过

```
1. 打开相应日期的日志文件
2. 在检查执行记录表中搜索 UI 名称
3. 查看该 UI 的初判类型和实际类型是否一致
```

### 场景 3：追踪某个问题的修复进展

```
1. 在问题记录表中找到该问题
2. 记录报告时间和问题内容
3. 后续再检查该 UI 是否仍有问题
4. 在备注中记录修复状态
```

### 场景 4：分析一周的检查趋势

```
1. 打开本周各日期的日志文件
2. 对比通过率的变化
3. 分析问题数是增加还是减少
4. 评估 Wiki 质量的改进情况
```
